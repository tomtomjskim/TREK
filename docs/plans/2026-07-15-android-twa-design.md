# Android TWA Distribution Design

## Objective

Provide a TREK-owned Android package that targets Android API 35 and opens
`https://trek.jsnetworkcorp.com` as a Trusted Web Activity (TWA). This removes
TREK's dependency on Samsung Internet's browser-generated WebAPK target SDK for
the owned install path while preserving the existing PWA.

## Product contract

- The owned Android application ID is `com.jsnetworkcorp.trek`.
- The first release uses version code `1`, version name `3.3.0`, target SDK 35,
  and compile SDK 36.
- The application opens the canonical HTTPS origin at `/` and stays within the
  `/` scope. Browser fallback remains available if domain verification fails.
- Existing browser PWA installation continues to work. Samsung Internet may
  still generate its own WebAPK and show an old-target warning; TREK cannot
  change that browser-owned package. The warning-free path is the TREK-signed
  APK.
- The web manifest gets a stable `id: '/'`, Korean language metadata, and keeps
  the existing name, icons, colors, scope, and start URL.
- The signed APK is publicly downloadable from
  `/downloads/trek-android.apk`. This is deliberate because Android package
  installation cannot depend on a TREK login session.

## Options considered

1. Keep browser-generated WebAPK installation only. Rejected as the primary
   solution: its package and target SDK are controlled by the browser vendor,
   not by TREK's web manifest or server.
2. Build and sign an owned TWA with Bubblewrap. Chosen: it preserves the web
   application as the product surface, has a small Android shell, supports
   Digital Asset Links, and gives TREK control of target SDK and signing.
3. Add Capacitor or a custom native WebView shell. Rejected for this increment:
   it adds a second runtime, native permission and lifecycle code, a larger
   security surface, and ongoing platform maintenance without a current native
   feature requirement.

## Web and Android identity

The PWA manifest identity is stable at `/`. `lang: 'ko'` describes the default
install metadata; in-app locale remains user-controlled.

The TWA is generated by the pinned Bubblewrap CLI `1.24.1` and committed as a
reproducible Android source project under `android/twa/`. The generated Gradle
configuration must retain:

- `applicationId "com.jsnetworkcorp.trek"`
- `targetSdkVersion 35`
- `compileSdkVersion 36`
- `versionCode 1`
- `versionName "3.3.0"`
- host `trek.jsnetworkcorp.com`, HTTPS start URL `/`

Generated release binaries, local SDK paths, Gradle caches, keystores, and
password files are not committed.

## Public route contract

The server registers Android release routes before the existing generic
`/.well-known/*` OAuth middleware. This ordering is required because that
middleware intentionally terminates unknown well-known paths.

### `GET /.well-known/assetlinks.json`

- Public, read-only, no cookie or bearer token required.
- Reads only the fixed file `assetlinks.json` from the Android release
  directory; no user-supplied path is accepted.
- Returns `application/json` and a short revalidation cache policy.
- The JSON grants only `delegate_permission/common.handle_all_urls` to
  `com.jsnetworkcorp.trek` and the release signing certificate's SHA-256
  fingerprint.
- Returns JSON 404 when the release document is absent or invalid.

### `GET /downloads/trek-android.apk`

- Public, read-only, no directory listing and no variable filename.
- Reads only the fixed file `trek-android.apk` from the same release directory.
- Returns `application/vnd.android.package-archive`, attachment disposition,
  and `no-store` so a replaced release is not served from an intermediary cache.
- Returns JSON 404 when the APK is absent.

The service worker navigation fallback excludes `/downloads/` as well as the
existing `/.well-known/` exclusion, preventing an APK navigation from being
replaced with the SPA HTML shell.

## Storage and signing boundary

- Runtime release files live at `/app/data/android` in the container, backed by
  `/mnt/oci-block-volume/services/trek/data/android` on the host.
- The long-lived signing material lives separately at
  `/mnt/oci-block-volume/services/trek/android-signing`.
- The signing directory is mode `0700`; keystore and password material are mode
  `0600`. The signing directory is never mounted into the TREK runtime
  container, copied into its image, committed, logged, or written to wiki.
- The Android SDK, Gradle cache, and build work directory use OCI block-volume
  paths so the root disk does not absorb the large toolchain.
- Public release files are atomically staged and then moved into the runtime
  release directory. The certificate fingerprint and artifact checksum are
  public verification data; passwords and private key bytes are not.

The signing key is the permanent update identity. Losing it prevents compatible
updates to the installed package. The block volume is persistent storage but is
not an independent backup; a separately encrypted offline backup remains an
operator follow-up before broad distribution.

## Authorization, privacy, and data impact

- No database, account, trip, registration, or admin permission behavior
  changes.
- The two release routes intentionally reveal only a public APK and the public
  certificate association document.
- Unknown well-known routes continue to return the existing JSON 404. OAuth/MCP
  discovery behavior and addon gates must not change.
- The TWA requests no TREK-specific Android runtime permissions. Website
  permissions continue to be mediated by the user's browser implementation.

## Build and release flow

1. Generate or reuse the release keystore at the fixed block-volume path.
2. Derive the public SHA-256 certificate fingerprint without printing secret
   material.
3. Generate/update the pinned Bubblewrap project and validate its configuration.
4. Build and sign APK/AAB with JDK and Android SDK caches on the block volume.
5. Verify signature, package ID, version, target SDK, permissions, and checksum.
6. Generate `assetlinks.json` from the verified signing certificate.
7. Publish the fixed APK and association document to the runtime release
   directory, deploy the web/server image, and verify both origin and Cloudflare
   edge responses.

## Failure and rollback

- Missing or malformed release files fail closed with 404 and do not fall back
  to SPA HTML.
- If Digital Asset Links is absent or mismatched, Android can open a browser UI
  instead of a verified fullscreen TWA; authentication and web data remain on
  the HTTPS origin.
- Web/server rollback is code and image only: restore the previous Docker image.
  Existing release files are inert to that image and may be moved out of the
  runtime directory if the download must be withdrawn.
- Android rollback is a new APK with a higher version code and the same signing
  key. An already-installed APK cannot be remotely removed by this self-hosted
  distribution.
- The signing key itself is not rolled back or rotated casually. Changing it
  creates a different update identity and requires a fresh package installation.

## Evidence plan

- Unit tests for both fixed public routes, headers, missing files, invalid
  `assetlinks.json`, and route registration before the generic well-known
  middleware.
- Bootstrap integration checks proving public responses and non-SPA 404s.
- Client test/build assertion for manifest `id`, `lang`, and download navigation
  exclusion.
- Static Android project verification for package, host, API levels, versions,
  permissions, and ignored secret/build outputs.
- Signed APK inspection with Android build tools and signature verification.
- Public HTTPS checks for health, manifest, Digital Asset Links, and APK headers
  plus Docker health/restart/log checks.
- A physical Samsung/Android install remains the final device evidence because
  no Android device is attached to this server.
