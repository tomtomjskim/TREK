# TREK v3.4.x Upstream Integration Design

> 작성일: 2026-07-19
> 상태: v3.4.1 증분 통합·운영 배포 완료
> 공식 기준: `liketrek/TREK` `v3.4.0` (`3ca1ef3`) + `v3.4.1` (`a099465`)
> 포크 기준: `tomtomjskim/TREK` `0367542`

## Goal

공식 v3.4.0 기능과 보안 수정사항을 현재 배포 포크에 통합하면서 지도 locale,
Fold responsive controls, Google 비용 안전장치, packing privacy/template scope, Android
TWA를 보존한다. 이번 통합으로 공식 migration 번호와 포크 migration 번호를 분리해
다음 공식 release도 같은 절차로 받을 수 있게 한다.

운영 배포, 공식 PR 생성, 개인 템플릿 R2 writer 활성화는 이번 단계의 목표가 아니다.

## v3.4.1 incremental contract

- 병합 대상은 움직이는 `upstream/main`이 아니라 공식 `v3.4.1`의 정확한 commit
  `a0994658890eae96624fb9cbe7f55867f047fea2`로 고정한다.
- tag target은 lightweight commit이며 최종 version-bump commit은 unsigned다. 실제
  기능 merge commit `91095ef`의 GitHub verification과 공식 CI 성공을 보조 증거로
  사용하되 signed tag라고 기록하지 않는다.
- 변경은 transit arrive-by 정렬, AirTrail hook 순서, 숙소 check-in-day 경로,
  per-user ntfy privacy, Synology streaming/logging, Atlas 및 plugin registry에 한정된다.
  공식 DB migration과 포크 migration, Google 비용 경계, packing privacy, Fold 지도
  controls는 변경하지 않는다.
- 3.4.1은 DB schema를 바꾸지 않으므로 증분 rollback class는 `code_only`다. 운영
  교체 전 SQLite online backup은 방어적 안전망으로 생성하며 즉시 rollback은 현재
  healthy 3.4.0 R2 image로 수행한다.
- auth/privacy 및 운영 배포가 포함되므로 gate는 계속 `full_gate_required`다.

## Evidence already fixed

- 포크 기준선: shared 32 files/138 tests, server 295 files/5,161 tests, client
  195 files/3,240 pass와 38 skip이 통과했다.
- 공식 `v3.4.0`과 포크는 v3.3.0 이후 각각 8/19 commits로 갈라졌다.
- 변경 파일 overlap은 86개이며 merge-tree에서 content conflict 31개가 확인됐다.
- 포크의 공식 migration 기준은 171이고, 로컬 172/173이 공식 v3.4의 172/173과
  충돌한다. 현재 운영 DB는 포크 schema version 173이다.

## Options considered

### A. 포크 migration을 공식 배열 뒤 176/177로 재배치

이번 upgrade만 가장 빨리 통과하지만 이미 운영 DB가 173이므로 별도 bridge가 필요하고,
다음 공식 release가 176/177을 추가하면 같은 충돌이 반복된다. 장기 구조로 채택하지 않는다.

### B. 공식/포크 migration track 분리 — selected

공식 `schema_version`은 그대로 유지하고 포크는 안정적인 문자열 ID를 가진 별도 이력을
사용한다. 공식 migration 파일을 upstream-owned surface로 되돌려 향후 merge conflict를
줄이고, runtime wrapper가 두 track의 순서와 legacy bridge를 관리한다.

### C. 모든 포크 기능을 즉시 plugin으로 이동

Google provider/usage 일부에는 적합하지만 공통 auth/privacy, packing core schema,
WebSocket visibility와 map layout은 현재 SDK 경계 밖이다. 선택 B와 병행해 가능한 기능만
점진적으로 plugin 또는 공식 PR로 이동한다.

## Target architecture

```text
database startup
  -> detect exact legacy v3.3 fork collision
  -> run official migrations (upstream schema_version)
  -> run fork migrations (fork_schema_migrations, stable string ids)
  -> assert fork invariants
  -> seed/start application
```

| 파일 | 소유권 | 책임 |
| --- | --- | --- |
| `server/src/db/migrations.ts` | upstream | 공식 migration과 숫자 version |
| `server/src/db/forkMigrations.ts` | fork | `jsnetworkcorp.*` migration, schema assertion, legacy signature |
| `server/src/db/migrationRunner.ts` | fork adapter | bridge → upstream → fork 실행 순서 |
| `server/src/db/database.ts` | integration seam | adapter만 호출 |

공식 migration 파일을 직접 수정하지 않는 것이 목표다. 애플리케이션·통합 테스트는
adapter를 사용하고 공식 migration 자체를 검증하는 테스트만 공식 runner를 직접 사용할
수 있다.

## Legacy collision state matrix

| 입력 상태 | 처리 | 기대 결과 |
| --- | --- | --- |
| stock/fresh schema 0..171 | 공식 migration 후 포크 migration | official 175 + fork IDs 2개 |
| 현재 포크 172, usage만 적용 | exact signature 확인 후 official marker 171로 정상화 | official 175 + 두 fork migration |
| 현재 포크 173, usage+packing 적용 | exact signature와 packing invariant 확인 후 marker 171로 정상화 | official 175 + 기존 데이터 보존 + fork IDs 2개 |
| stock 공식 172/173 partial | 공식 plugin column signature로 구분 | 남은 공식 migration 후 fork migration |
| 공식 173 schema commit 후 marker가 172에 머문 crash window | 공식 172 signature + `trek_range`를 함께 확인 | 공식 173을 idempotent 재실행한 뒤 175까지 진행 |
| legacy bridge 기록 후 같은 crash window | bridge ID + fork invariant + 공식 ahead signature 확인 | bridge를 반복하지 않고 공식 173부터 재개 |
| stock 공식 175 | bridge 없음 | fork migration만 적용 |
| 이미 통합된 DB 재실행 | 모든 migration skip, invariant 재검증 | 변경 없음 |
| 로컬/공식 marker가 섞인 unknown partial | 자동 추정 금지 | startup fail closed |

Legacy marker 정상화와 bridge history 기록은 한 transaction에서 수행한다. 이후 공식
migration의 schema transaction이 commit됐지만 숫자 marker 갱신 전에 프로세스가 중단된
경우도 exact ahead signature에서만 재개한다. stock과 bridge state를 별도로 검증하고,
그 외 mixed/partial state는 자동 추정하지 않고 fail closed한다. 포크 migration은 기존
schema를 검증한 뒤 누락된 ID만 기록한다.

## Architecture readiness

- Gate: `full_gate_required`
- Surfaces: DB/schema, auth/packing privacy, provider cost/rate limit, plugin capability,
  maps/settings UI, Android static release, Docker/Compose, public HTTPS
- Rollback: `data_migration`
- Main risk: migration number collision로 공식 plugin columns가 누락되는 것, packing
  privacy 완화, Google hard cap 우회, 지도 initial-fit와 locale 충돌, Fold control 회귀
- Rollback floor: 배포 전 SQLite online backup + 현재 immutable image
  `trek:3.3.0-packing-template-r1-0367542`

## Conflict resolution contract

- Maps: upstream initial-fit/touch fixes와 local label expression/adaptive controls를 함께 유지한다.
- Packing: upstream UI/plugin write 기능을 받아들이되 Personal/Shared owner-only 변경,
  viewer-filtered reads와 instance-only template writer를 완화하지 않는다.
- Google: upstream maps/admin 변화와 local usage reservation/hard cap/enrichment switch를
  함께 유지한다. provider call 전 durable reservation 순서를 보존한다.
- Settings/i18n: upstream `DEFAULT_SETTINGS` export와 `map_label_language`를 합치고 모든
  locale key parity를 유지한다.
- Deployment: 공식 v3.4 코드/version을 사용하되 Docker image reference는 소문자이고,
  JSNetworkCorp TWA와 운영 override는 instance-only로 유지한다.

## Evidence plan

1. migration unit tests: state matrix 전부, row count/FK/invariant, 재실행, unknown partial fail closed.
2. focused server tests: maps usage cap/enrichment, packing REST/MCP/plugin/privacy, admin auth.
3. focused client tests: map labels/initial fit, adaptive controls, packing rows, admin usage.
4. strict i18n parity, 세 workspace typecheck와 production build.
5. 전체 `npm test`와 공식 coverage gate에 상응하는 회귀.
6. 운영 DB의 민감 데이터가 없는 사본으로 backup→migration→integrity/FK→재실행→restore rehearsal.
7. local immutable ARM64 image와 Compose config dry run. 운영 container 교체는 별도 승인 뒤 수행.

## Upstream contribution strategy

지도 locale, Fold controls, packing privacy처럼 일반적인 변경은 각각 `upstream/dev`에서
한 기능씩 재구성한다. Discord 승인과 issue/discussion 연결 전에는 PR을 만들지 않는다.
Google/Android/infra처럼 instance 결합이 강한 부분은 core에서 최소화하고 plugin 또는
instance-only lane에 둔다. 공식 release가 기능을 포함하면 patch inventory의 retirement
조건에 따라 로컬 구현을 제거한다.

구현·검증 결과와 남은 운영 gate는
`docs/plans/2026-07-19-upstream-v3.4-integration-evidence.md`에 기록한다.
